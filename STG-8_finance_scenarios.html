Option A — Ops-only model
MF scope: limited Finance: stays in Blue10
What this option means

MyFreight remains an Ops tool. Finance does not log in to MF. Ops records expected costs at the right moment in the shipment lifecycle, and Finance receives that data via Blue10 or an export — outside of MF. Invoice matching and approval continue as today, but now with a structured accrual to match against.

How the process works
1
Milestone fires in MF
ATD (origin incoterms) or ATA (destination incoterms) confirmed by MF live tracking. Accrual task automatically set to Pending on the shipment.
↓
2
Ops records expected costs
Ops opens the Accrual task and logs: supplier, shipment reference, charge code, expected amount, currency. Task set to Completed.
↓
3
Agent invoice received
Invoice document arrives via agent API. MF auto-forwards to Blue10 and marks Debit Note Received task as Completed.
↓
4
Finance matches in Blue10
Finance compares invoice against the accrual data received via export or Blue10 integration. Discrepancies resolved manually between Finance and Ops — outside MF.
↓
5
Shipment closure gate
Shipment cannot be closed in MF while Accrual task is Pending. Ops must complete the accrual before the shipment is marked done.
Tradeoffs
What this gains
Simpler scope and faster to implement. No Finance user provisioning needed. No new MF team logic. Accrual data is structured and available to Finance for the first time — a significant improvement on today's state.
What this costs
Finance has no live visibility inside MF. Disputes and discrepancies are handled manually outside the system with no audit trail in MF. The loop between accrual and invoice resolution is not closed in one place.
Decisions needed
1
Do we proceed with Option A, or do we move to Option B?
Decision owner: Finance + Ops + IT
2
ATD trigger for origin incoterms — use ATD, or default to ATA for all?
Decision owner: Dev assessment + Finance/Ops sign-off
3
How does Finance receive accrual data from MF — export, report, or Blue10 integration?
Decision owner: Finance + IT
4
Shipment closure gate — hard block or soft warning?
Decision owner: Dev assessment + Ops
5
Charge code list — who maintains it and where?
Decision owner: Finance + Forwarding

Option B — Finance active model
MF scope: extended Finance: works in MF
What this option means

Finance users are provisioned in MyFreight and work actively within a dedicated Financial Handling team. Ops records the accrual; Finance picks up from there — invoice verification, matching, approval or dispute — all inside MF. The full financial loop is closed in one system, with a complete audit trail.

How the process works
1
Milestone fires in MF
ATD (origin incoterms) or ATA (destination incoterms) confirmed by MF live tracking. Accrual task automatically set to Pending on the shipment.
↓
2
Ops records expected costs
Ops opens the Accrual task and logs: supplier, shipment reference, charge code, expected amount, currency. Task set to Completed — visible to Finance in the Financial Handling team filter.
↓
3
Invoice received — auto-forwarded to Blue10
Agent invoice arrives, MF forwards to Blue10 and creates Invoice Verification task set to Pending. Task is held if Accrual task is not yet Completed.
↓
4
Finance verifies in MF
Finance opens Invoice Verification task. Compares invoice against accrual. Match → Invoice Approved. Mismatch → Dispute task created.
↓
5
Dispute resolution (if needed)
Finance and Ops resolve discrepancy. Outcome logged in MF — credit note raised, amount adjusted, or invoice accepted as is. Dispute task Completed.
↓
6
Shipment closure gate
Shipment cannot be closed while Accrual or Dispute task is Pending. Full financial loop must be closed before shipment is marked done.
Tradeoffs
What this gains
Full auditability end-to-end. Disputes structured and traceable in MF. No data leaving the system for financial steps. Sets up BC integration (2027) cleanly. Finance has live visibility of accrual status on every shipment.
What this costs
Finance user provisioning and training. New Financial Handling team in MF with access controls. Significant development effort — trigger logic, status transitions, task sequencing, dispute handling. Error-sensitive; automations are required, not optional.
Decisions needed
1
Do Finance users get full MF access or read-only outside their tasks?
Decision owner: Finance + IT
2
ATD trigger for origin incoterms — use ATD, or default to ATA for all?
Decision owner: Dev assessment + Finance/Ops sign-off
3
Invoice Verification gate — hold until accrual is complete, or allow with warning?
Decision owner: Finance + Dev
4
Invoice Approved — implicit on match, or explicit Finance approval step required?
Decision owner: Finance
5
Shipment closure gate — hard block or soft warning?
Decision owner: Dev assessment + Ops
6
Charge code list — who maintains it and where?
Decision owner: Finance + Forwarding
